# AI Collaboration Log

> This document records how AI assistance was used to build the Fusion Feedback Service.

## Tool Used

**Claude (Anthropic)** — used as the primary coding assistant for architecture design, code generation, and refinement throughout the session.

## Task Breakdown & Prompts Used

### Task 1 — Architecture & Project Scaffold

**Mission:** Design a clean Node.js project structure that covers all requirements without overbuilding.

**Prompt used:**
> "Design a Node.js Express service that accepts free-text feedback, persists it to SQLite with status tracking (RECEIVED → ANALYZING → DONE | FAILED), runs async LLM analysis in an in-process queue, validates the JSON output with Zod, and supports hash-based deduplication. The service should be minimal — no Docker, no Redis, no ORM. Show the file structure and the responsibility of each module."

**What the AI produced:** A 6-file structure (`app.js`, `db.js`, `queue.js`, `aiService.js`, `feedbackService.js`, `routes.js`) with clear separation of concerns. The AI correctly identified that `setImmediate` should be used for non-blocking queue dispatch.

---

### Task 2 — Database Layer

**Mission:** Build a SQLite layer with WAL mode, prepared statements, and a schema that captures all required fields plus the deduplication hash.

**Prompt used:**
> "Write a `db.js` module using `better-sqlite3`. It should: (1) open/create a SQLite database with WAL mode on first call, (2) run a migration to create the feedback table if it doesn't exist, (3) expose insert, findByHash, findById, listAll, updateStatus, and updateAnalysis as named exports using prepared statements. The table needs: id, content, content_hash (UNIQUE), status (CHECK constraint), raw_ai_response, analysis, created_at, updated_at."

**What the AI produced:** Correct implementation with lazy-loaded prepared statements and a singleton DB handle. The CHECK constraint on status was included, which is good defensive practice.

---

### Task 3 — AI Service (Ollama Integration)

**Mission:** Call Ollama's local API, force JSON output, and validate it with Zod.

**Prompt used:**
> "Write an `aiService.js` module that: (1) calls Ollama's `/api/chat` endpoint with the `gemma4:e4b` model, (2) uses a strict system prompt that forces pure JSON output with no markdown and no preamble, (3) defensively strips accidental code fences from the response, (4) parses the result with JSON.parse and validates it with Zod against this schema: `{ sentiment: enum[positive,neutral,negative], feature_requests: [{ title: string, confidence: float 0-1 }], actionable_insight: string }`, (5) throws a descriptive error on any failure."

**What the AI produced:** Working implementation with `fetch`, Zod schema, and fence-stripping. Initial system prompt was too verbose (see correction below).

---

### Task 4 — In-Process Queue

**Mission:** Implement a non-blocking async queue that processes feedback items with DB state updates.

**Prompt used:**
> "Write a lightweight in-process async FIFO queue in Node.js with no external dependencies. It should: support configurable concurrency (default 2), update the DB status to ANALYZING before calling the AI, then to DONE or FAILED depending on the outcome, persist raw AI response and validated analysis, log errors without crashing the process. Use setImmediate in the caller so submission is always non-blocking."

**What the AI produced:** Clean implementation using a simple array as the queue, a `drain()` loop, and `.finally()` for concurrency accounting. Exactly the right level of complexity for the scope.

---

### Task 5 — Routes & Error Handling

**Mission:** Wire up Express routes with proper HTTP semantics.

**Prompt used:**
> "Write Express routes for: POST /feedback (202 on success, 409 on duplicate, 400 on missing content), GET /feedback (with optional ?status, ?limit, ?offset query params), GET /feedback/:id (404 if not found), POST /feedback/:id/retry (409 if not FAILED, 404 if not found). Keep it thin — all business logic in feedbackService."

**What the AI produced:** Correct routes. One refinement was needed: the initial draft used `req.body.content` without a null-guard, which would throw if `Content-Type` header was missing. Fixed by adding `req.body?.content` and an explicit type check.

## Where AI Output Was Wrong — and How I Corrected It

### Problem: Ollama returning non-JSON output

The initial system prompt generated by the AI was written in a conversational style:

> *"You are a helpful product analyst. Please analyze the following feedback and return a JSON object with the fields: sentiment, feature_requests (an array of objects with title and confidence), and actionable_insight. Make sure your response is valid JSON."*

**What went wrong:** `gemma4:e4b` mirrored the polite, conversational tone and prefixed its JSON output with text like:
> *"Sure! Here is my analysis of the feedback:"*
> *```json*  
> *{ ... }*

This caused `JSON.parse()` to throw on every call.

**How I corrected it:**

1. **Rewrote the system prompt** to be terse and command-style with explicit negative instructions:
   > *"respond with ONLY a valid JSON object — no markdown, no code fences, no explanation"*

2. **Added a post-processing fence-stripper** as a defensive fallback, in case the model still wraps output in ` ```json ``` `:
   ```js
   const cleaned = raw.replace(/^```(?:json)?\s*/i, '').replace(/\s*```$/, '').trim();
   ```

3. **Lowered temperature to 0.1** to reduce creative/verbose tendencies in the model.

**Lesson:** Small local models need more directive, constrained prompts than large cloud models. Defensive parsing is non-negotiable.

## What I Would Improve With More Time

- **Durable queue:** Replace the in-memory queue with BullMQ + Redis so enqueued jobs survive process restarts.

- **Structured logging:** Add `pino` with request IDs so each feedback item's lifecycle is traceable across log lines.

- **Automatic retry with backoff:** Currently retry is manual (via API). An exponential backoff strategy inside the queue would handle transient Ollama failures without user intervention.

- **Ollama health check on boot:** Ping Ollama before accepting traffic; fail fast with a clear error message if the model isn't loaded.

- **Integration tests:** A Jest suite with `supertest` + in-memory SQLite + a mock Ollama server to exercise the full state machine deterministically.

- **Smarter deduplication:** The current hash normalises casing and whitespace. Semantic deduplication (embedding-based similarity) would catch near-duplicates like "add dark mode" vs "please add a dark theme".
